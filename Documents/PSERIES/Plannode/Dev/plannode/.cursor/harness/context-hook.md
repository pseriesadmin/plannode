# context-hook.md — Plannode 컨텍스트 드리프트 방지 훅
# 위치: .cursor/harness/context-hook.md
# 목적: 장시간 세션에서 AI 드리프트·파일럿 갭 망각 방지
# 적용: harness-executor가 아래 상황 감지 시 자동 실행

---

## 컨텍스트 드리프트 감지 조건

아래 중 하나라도 해당되면 즉시 훅을 실행한다.
구현 코드를 먼저 작성하지 않는다.

```
HOOK-1: 이전 작업 참조 요청
  트리거: "아까", "이전에", "방금 전", "앞에서 만든", "기존 로직",
          "저번에", "같은 방식으로", "참고해서"

HOOK-2: 파일럿 로직 재확인 요청
  트리거: "파일럿에서", "plannode.js에서", "원래 어떻게",
          "기존 렌더링 방식", "lm 계산이 어떻게", "applyTx 방식"

HOOK-3: 파일럿 갭 관련 작업
  트리거: "transform", "SVG 간선", "노드 드래그", "줌", "패닝",
          "parent_id", "addChild", "openProj", "루트 노드",
          "canvasContainer", "bind:this"

HOOK-4: 새 채팅 재시작 후 첫 작업
  트리거: TASK.md 로드 직후 첫 번째 NOW 태스크 실행 시
```

---

## HOOK-1 — 이전 작업 참조 시

> 패턴: AI가 없는 내용을 있다고 만들어내는 할루시네이션

```
실행 순서:

1. 구현 중단 선언
   "⏸ HOOK-1: 이전 작업 참조 요청 감지. 파일 확인 후 진행합니다."

2. 실제 파일에서 확인 (Read 툴 사용)
   → TASK.md DONE 항목에서 완료된 작업 목록 확인
   → GSD_LOG.md에서 실제 구현된 파일 경로 확인
   → 해당 파일을 직접 Read해서 실제 코드 확인

3. 확인 결과 출력 (1~3줄)
   "확인된 내용: [실제 파일에서 읽은 것]"
   "확인 안 됨: [파일에 없는 것 — 구현 안 됐거나 다른 위치]"

4. Stephen 확인 요청 (애매한 경우만)
   "위 내용으로 진행할까요? 다른 파일 확인이 필요하면 경로 알려주세요."

금지:
❌ 파일 확인 없이 "아까 이렇게 했으니까" 추측으로 진행
❌ "기억에 의하면" 표현 사용
❌ 존재하지 않는 함수명·파일 경로 생성
```

---

## HOOK-2 — 파일럿 로직 재확인 시

> 패턴: 파일럿 원본과 다른 방식으로 SvelteKit 재구현

```
실행 순서:

1. 구현 중단 선언
   "⏸ HOOK-2: 파일럿 로직 확인 요청 감지. 원본 파일을 읽겠습니다."

2. 파일럿 소스 파일 직접 Read
   우선순위 순서:
   → plannode.js (해당 함수·라인)
   → index.html (DOM 구조 확인)
   → docs/PILOT_FUNCTIONAL_SPEC.md (해당 섹션)

3. 읽은 내용 기반으로 파일럿 동작 요약
   "파일럿 기준 동작: [plannode.js/index.html에서 읽은 것]
    - 방식: [실제 코드에서 읽은 것]
    - 관련 변수·함수: [실제 코드에서 읽은 것]"

4. SvelteKit 구현과 충돌 여부 확인
   "이번 NOW 태스크와 충돌하는 부분: [있음/없음]"
   충돌 있으면 → Stephen에게 보고 후 대기
   충돌 없으면 → 바로 진행

금지:
❌ 파일럿 로직을 "기억"으로 재구성
❌ plannode.js 읽지 않고 "이렇게 돼있을 거예요" 추측
❌ 파일럿과 다른 방식으로 조용히 재구현
```

## HOOK-3 — 파일럿 갭 관련 작업 시

> 패턴: 갭 분석 체크 없이 잘못된 방식으로 SvelteKit 구현

```
실행 순서:

1. 구현 전 체크리스트 출력 (코드 작성 전에 반드시)
   "⏸ HOOK-3: 파일럿 갭 관련 작업 감지. 갭 체크 먼저 합니다."

2. 관련 갭 항목 확인
   → docs/PILOT_FUNCTIONAL_SPEC.md §9 테이블에서 해당 영역 확인
   → AGENTS.md 주의 강화 체크 목록 확인
   
3. 파일럿과 SvelteKit 현황 대조 출력
   "파일럿 동작: [§9 테이블에서 읽은 파일럿 방식]
    SvelteKit 현황: [§9 테이블에서 읽은 현재 상태]
    리스크: [§9 테이블에서 읽은 위험 항목]"

4. 올바른 구현 방향 확인 후 진행
   "이번 NOW 태스크에서 적용할 파일럿 방식: [확인된 내용]
    이 방향으로 진행합니다."

금지:
❌ 갭 항목 확인 없이 SvelteKit 방식대로 구현
❌ SVG를 transform 컨테이너 밖에 배치
❌ parent_id에 프로젝트 id 사용
❌ nodes 배열 빈 배열 저장 후 루트 미생성
```

## HOOK-4 — 새 채팅 재시작 후 첫 작업

> 패턴: 이전 세션 맥락 없이 시작해서 방향 이탈

```
새 채팅에서 TASK.md 로드 직후 아래를 반드시 실행한다.
TASK.md만 읽고 바로 구현 시작하지 않는다.

실행 순서:

1. 현재 상태 파악 (Read 툴)
   → TASK.md: DONE / NOW / NEXT / BLOCKED 현황
   → GSD_LOG.md: 마지막 완료 태스크와 파일
   → plan-output.md: 이번 사이클 아젠다 원문

2. 재시작 브리핑 출력
   "📋 재시작 브리핑
    아젠다: [plan-output.md 첫 줄]
    완료: [DONE 태스크 수]개
    현재 NOW: [태스크명]
    마지막 수정 파일: [GSD_LOG 마지막 항목]
    이어서 진행합니다."

3. NOW 태스크 도메인 판별 (GSD / GSD+)
   → GSD+ 해당 시 파일럿 갭 체크 먼저 수행

금지:
❌ TASK.md만 보고 바로 구현 시작
❌ GSD_LOG 확인 없이 "처음부터" 시작
❌ 이전 세션 내용을 "기억"으로 채우기
```

## 컨텍스트 드리프트 의심 시 Stephen 입력 명령

```
"컨텍스트 체크"   → harness-executor가 HOOK-4 재시작 브리핑 실행
"파일 확인해"     → HOOK-1 실행 (추측 없이 실제 파일 Read)
"파일럿 읽어"     → HOOK-2 실행 (plannode.js 직접 Read)
"갭 확인해"       → HOOK-3 실행 (PILOT_FUNCTIONAL_SPEC.md §9 확인)
```
