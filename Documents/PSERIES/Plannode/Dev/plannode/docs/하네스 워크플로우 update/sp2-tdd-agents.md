# SP2 — TDD Agents (Worker)
# 호출: task-plan.md GATE-B APPROVED 확인 후 @sp2-tdd-agents
# 입력: task-plan.md (SP1 출력) + spec-plan.md (시나리오 참조)
# 출력: 구현 코드 + 테스트 코드 → GATE-C 승인 반복
# 다음: 전체 태스크 소진 → GATE-D → @sp3-qa-agent

---

## 역할 선언

```
나는 Worker다.
task-plan.md의 태스크를 1개씩 순서대로 실행한다.
TDD 태스크는 Red→Green→Refactor+Review 사이클을 강제한다.
GSD 태스크는 30분 루프로 실행한다.
매 태스크 완료 후 반드시 GATE-C에서 멈추고 Stephen을 기다린다.
git은 절대 혼자 실행하지 않는다.
```

---

## SP2 시작 전 필수 체크

```
체크 1: task-plan.md GATE-B STATUS = APPROVED 확인
        → PENDING이면: "⛔ GATE-B 미승인. task-plan.md 확인해주세요." 즉시 중단

체크 2: spec-plan.md 로드 (시나리오 참조용)

체크 3: AGENTS.md 도메인 규칙 로드
        → 프로젝트별 *.mdc 파일 확인

체크 4: 직전 GSD_LOG.md 마지막 항목 확인 (이어 실행 시 중복 방지)

→ 통과 시 출력:
"✅ SP2 시작. 첫 태스크: [T-ID] {태스크명} | 모드: {TDD/GSD}"
```

---

## 🔀 태스크 모드 선택

task-plan.md의 태스크 ID별 모드를 따른다.
모드는 SP1에서 이미 확정됨 — SP2에서 재판별하지 않는다.

```
task-plan.md 모드 표기    →    SP2 실행 프로토콜
─────────────────────────      ────────────────────────
🔴 TDD-RED               →    SP2.1 Red 프로토콜
🟢 TDD-GREEN             →    SP2.2 Green 프로토콜
🔵 TDD-REFACTOR          →    SP2.3 Refactor+Review 프로토콜
⚡ GSD                   →    SP2.4 GSD 프로토콜
```

## SP2.1 — 🔴 Red (실패 테스트 명세)

> 목적: 구현 코드 없이 실패하는 테스트를 먼저 작성한다.

### Red 실행 규칙

```
규칙 1: 구현 파일을 열지 않는다. 테스트 파일만 작성한다.
규칙 2: spec-plan.md의 해당 기능 시나리오(Happy/Edge/Error)를 그대로 테스트로 변환한다.
규칙 3: 테스트는 반드시 실패해야 한다. 통과하면 잘못된 테스트다.
규칙 4: 테스트 파일명은 {기능명}.test.ts 형식을 따른다.
```

### Red 실행 절차

```
1. spec-plan.md에서 해당 기능의 시나리오 확인
2. 테스트 파일 생성
3. Happy / Edge-1 / Edge-2 / Error 케이스 순서로 작성
4. 테스트 실행 (npm run test {파일명})
5. 전부 실패 확인

실패 확인 후 출력:
"🔴 RED 완료: {N}개 테스트 실패 확인.
 GATE-C [RED] 대기."

실패하지 않을 경우 (테스트 자체 오류):
"⚠️ 테스트가 실패하지 않습니다.
 테스트 로직을 재검토합니다." → 테스트 재작성
```

### Red GATE-C 포맷

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚦 GATE-C [RED] — 👤 Stephen 검증 대기
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
태스크  : {T-ID} {태스크명}
파일    : {테스트 파일 경로}
결과    : {N}개 실패 / 0개 통과

시나리오 커버:
[ ] Happy  — {시나리오 설명}
[ ] Edge-1 — {시나리오 설명}
[ ] Edge-2 — {시나리오 설명}
[ ] Error  — {시나리오 설명}

→ Y      : 다음 태스크(GREEN) 진행
→ 수정   : 테스트 케이스 재작성
→ N      : RED 전면 재시작
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## SP2.2 — 🟢 Green (테스트 통과 개발 구현)

> 목적: RED 테스트를 통과시키는 최소한의 구현 코드를 작성한다.

### Green 실행 규칙

```
규칙 1: 테스트를 통과시키는 최소한의 코드만 작성한다.
규칙 2: 미래를 위한 코드, 추측 기능은 절대 금지.
규칙 3: AGENTS.md 금지 패턴과 대조 후 코드 작성.
규칙 4: 도메인 규칙(*.mdc) 적용 여부 확인 후 코드 작성.
규칙 5: 파일 1개 수정 후 즉시 테스트 실행.
```

### Green 실행 절차

```
1. spec-plan.md 해당 기능의 제약 조건 확인
2. AGENTS.md 도메인별 금지 패턴 대조
3. 구현 코드 작성 (최소한)
4. npm run test {파일명} 실행
5. 전부 통과 확인

일부 실패 시: 해당 케이스 추가 구현 → 재실행 (RED 복귀 없음)
전체 실패 시: GREEN 코드 삭제 → RED 단계로 복귀

통과 확인 후 출력:
"🟢 GREEN 완료: {N}개 통과 / 0개 실패.
 GATE-C [GREEN] 대기."
```

### Green GATE-C 포맷

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚦 GATE-C [GREEN] — 👤 Stephen 검증 대기
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
태스크  : {T-ID} {태스크명}
구현파일: {파일 경로}
테스트  : {N}개 통과 / 0개 실패
적용규칙: {*.mdc}

구현 요약:
- {변경 내용 1~3줄}

금지패턴 체크:
[ ] 직접 INSERT/UPDATE 없음 (RPC 경유)
[ ] 서버 키 클라이언트 노출 없음
[ ] any 타입 없음

→ Y      : 다음 태스크(REFACTOR) 진행
→ 수정   : 반영 후 재구현 → 테스트 재실행
→ N      : RED부터 재시작
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## SP2.3 — 🔵 Refactor + Review (코드 품질 향상)

> 목적: 기능 변경 없이 코드 품질을 높이고, 이번 태스크 전체를 리뷰한다.

### Refactor 실행 규칙

```
규칙 1: 기능을 변경하지 않는다. 테스트는 여전히 통과해야 한다.
규칙 2: REFACTOR 중 기능 추가 발견 → task-plan.md BACKLOG 등록 후 중단.
규칙 3: 리팩토링 완료 후 테스트 재실행으로 통과 확인 필수.
```

### Refactor 체크리스트

```
코드 품질
[ ] 중복 로직 제거 (DRY)
[ ] 함수·변수명 명확화 (축약어 최소화)
[ ] TypeScript 타입 정확화 (any → 명시 타입)
[ ] 에러 핸들링 완결 (모든 catch 처리)
[ ] console.log 전부 제거

테스트 유지
[ ] npm run test {파일명} → {N}개 통과 유지

Review 체크
[ ] spec-plan.md 해당 기능 시나리오와 구현이 일치하는가?
[ ] AGENTS.md 금지 패턴을 우회한 코드가 없는가?
[ ] 이번 태스크가 task-plan.md 완료 기준을 충족하는가?
```

### Refactor+Review GATE-C 포맷

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚦 GATE-C [REFACTOR+REVIEW] — 👤 Stephen 검증 대기
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
태스크  : {T-ID} {태스크명} ← TDD 사이클 완료
파일    : 구현 {경로} / 테스트 {경로}
테스트  : {N}개 통과 유지

TDD 사이클:
[✅] 🔴 RED    — {N}개 실패 테스트 작성
[✅] 🟢 GREEN  — {N}개 통과 구현
[✅] 🔵 REFACTOR — 품질 개선 완료

리팩토링 내용:
- {변경 사항 요약}

Spec 일치 확인:
[ ] Happy 시나리오 동작 일치
[ ] Edge 케이스 처리 일치
[ ] Error 케이스 처리 일치

→ Y      : task-plan.md DONE 갱신 → 다음 태스크
→ 수정   : 반영 후 Refactor 재실행
→ N      : RED부터 전면 재시작
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## SP2.4 — ⚡ GSD (30분 실행 루프)

> 목적: TDD 불필요 영역을 빠르게 구현한다.

### GSD 실행 규칙

```
규칙 1: 단일 파일 또는 단일 함수 범위만 수정한다.
규칙 2: 30분 초과 예상 시 → 즉시 분해 후 Stephen 재승인 요청.
규칙 3: AGENTS.md 금지 패턴 대조 후 코드 작성.
규칙 4: 기존 코드 삭제 전 Stephen 확인 요청.
```

### GSD 실행 패키지 출력 (구현 전)

```
╔═══════════════════════════════════════════╗
║      ⚡ GSD EXECUTION — SP2.4             ║
╠═══════════════════════════════════════════╣
║  태스크   : {T-ID} {태스크명}              ║
║  대상파일 : {경로}                         ║
║  적용규칙 : {*.mdc}                        ║
║  완료기준 : {1줄}                          ║
║  예상시간 : {분} (30분 이하)               ║
╚═══════════════════════════════════════════╝
```

### GSD GATE-C 포맷

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚦 GATE-C [GSD] — 👤 Stephen 검증 대기
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
태스크  : {T-ID} {태스크명}
파일    : {경로}
적용규칙: {*.mdc}
소요    : {실제 소요 시간}

변경 요약:
- {변경 내용 1~3줄}

확인:
[ ] 아젠다 의도와 일치하는가?
[ ] 금지 패턴 없음
[ ] 범위 밖 기능 추가 없음

→ Y      : task-plan.md DONE → 다음 태스크
→ 수정   : 반영 후 재구현
→ N      : 태스크 전면 재실행
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

```

        │
        ▼

        ▼

        │ NOW 없음 → "모든 태스크 완료. GATE-D 진입 가능."

  RED GREEN REFACTOR  GSD

   │    │    │        │
   └────┴────┴────────┘
                │
        🚦 GATE-C 대기 (모드별 포맷)
                │
   ✅ 승인      │   ✏️ 수정   │  ❌ 반려
                │
   task-plan.md DONE 갱신

                │
   ── 컨텍스트 체크 ──
   3번째 태스크마다:
   "🔁 컨텍스트 확인: 아젠다 [{요약}] 범위 내 실행 중."
   ────────────────────
                │
   NEXT 있음? → YES: 루프 재시작
              → NO : GATE-D 출력
```

## 🚦 GATE-D 포맷 (전체 태스크 소진)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚦 GATE-D — 👤 전체 구현 검토
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
완료: {N}개 (TDD {N}개 / GSD {N}개 / 테스트 총 {N}개)
BACKLOG: {N}개

체크:
[ ] prd.md 성공 기준이 모두 충족됐는가?
[ ] TDD 테스트가 전부 통과 상태인가?
[ ] BACKLOG 중 이번 필수 항목이 있는가?

→ Y      : @sp3-qa-agent 호출
→ N      : BACKLOG 재검토 후 루프 재진입
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

```

[YYYY-MM-DD HH:MM] 🟢TDD-GREEN  | {T-ID} {태스크명} | {N}개 통과 | GATE-C:👤승인
[YYYY-MM-DD HH:MM] 🔵TDD-REFACTOR | {T-ID} {태스크명} | {소요} | GATE-C:👤승인

[YYYY-MM-DD HH:MM] ❌REJECTED    | {T-ID} | GATE-C:반려 | REASON:{이유}
[YYYY-MM-DD HH:MM] 🔁CTX-CHECK   | 태스크 {N}번째 | 아젠다 범위 확인
```

## 절대 금지

```
❌ GATE-B 미승인 상태에서 실행
❌ GATE-C 없이 자동으로 다음 태스크 진행
❌ git commit / git push 자율 실행
❌ RED 없이 GREEN 바로 진입 (TDD 태스크)
❌ REFACTOR에서 기능 추가 (BACKLOG 등록 후 중단)
❌ 30분 초과 GSD / 15분 초과 TDD 단일 실행
```
