# context-hook.md — 크레이지샷 컨텍스트 롯 방지 훅
# 위치: .cursor/harness/context-hook.md
# 목적: 장시간 세션에서 AI 드리프트·컨텍스트 롯 발현 방지
# 적용: gsd-agent가 아래 상황 감지 시 자동 실행

---

## 컨텍스트 롯 감지 조건

아래 중 하나라도 해당되면 즉시 훅을 실행한다.
구현 코드를 먼저 작성하지 않는다.

```
HOOK-1: 이전 작업 참조 요청
  트리거: "아까", "이전에", "방금 전", "앞에서 만든", "기존 로직",
          "저번에", "같은 방식으로", "참고해서"

HOOK-2: 로직 재확인 요청
  트리거: "기억해", "어떻게 짰지", "로직이 뭐였지",
          "가격 계산 방식", "할인 적용 순서", "단가 구조"

HOOK-3: 디자인 시스템 적용
  트리거: "컴포넌트", "디자인", "UI", "레이아웃", "스타일",
          "화면", "버튼", "모달", "탭", "카드"

HOOK-4: 새 채팅 재시작 후 첫 작업
  트리거: TASK.md 로드 직후 첫 번째 NOW 태스크 실행 시
```

---

## HOOK-1 — 이전 작업 참조 시

> 패턴: AI가 없는 내용을 있다고 만들어내는 할루시네이션

```
실행 순서:

1. 구현 중단 선언
   "⏸ HOOK-1: 이전 작업 참조 요청 감지. 파일 확인 후 진행합니다."

2. 실제 파일에서 확인 (Read 툴 사용)
   → TASK.md DONE 항목에서 완료된 작업 목록 확인
   → GSD_LOG.md에서 실제 구현된 파일 경로 확인
   → 해당 파일을 직접 Read해서 실제 코드 확인

3. 확인 결과 출력 (1~3줄)
   "확인된 내용: [실제 파일에서 읽은 것]"
   "확인 안 됨: [파일에 없는 것 — 구현 안 됐거나 다른 위치]"

4. Stephen 확인 요청 (애매한 경우만)
   "위 내용으로 진행할까요? 다른 파일 확인이 필요하면 경로 알려주세요."

금지:
❌ 파일 확인 없이 "아까 이렇게 했으니까" 추측으로 진행
❌ "기억에 의하면" 표현 사용
❌ 존재하지 않는 함수명·파일 경로 생성
```

---

## HOOK-2 — 로직 재확인 요청 시

> 패턴: 초반에 짠 로직과 다른 방식으로 재구현

```
실행 순서:

1. 구현 중단 선언
   "⏸ HOOK-2: 로직 확인 요청 감지. 실제 구현 파일을 읽겠습니다."

2. 크레이지샷 핵심 로직 파일 직접 Read
   우선순위 순서:
   → supabase/functions/ 또는 src/lib/ 내 해당 로직 파일
   → calculate_final_price RPC 파일
   → atomic_reserve_asset RPC 파일
   → 할인 계산 관련 파일

3. 읽은 내용 기반으로 현재 구현 상태 요약
   "현재 구현된 로직: [파일명] 기준
    - 단가 적용 방식: [실제 코드에서 읽은 것]
    - 할인 적용 순서: [실제 코드에서 읽은 것]
    - 엣지케이스 처리: [실제 코드에서 읽은 것]"

4. 이번 태스크와 충돌 여부 확인
   "이번 NOW 태스크와 충돌하는 부분: [있음/없음]"
   충돌 있으면 → Stephen에게 보고 후 대기
   충돌 없으면 → 바로 진행

금지:
❌ 로직을 "기억"으로 재구성
❌ 파일 읽지 않고 "이렇게 돼있을 거예요" 추측
❌ 기존 로직과 다른 방식으로 조용히 재구현
```

## HOOK-3 — 디자인 시스템 적용 시

> 패턴: 규칙 문서 있어도 컴포넌트 혼용하거나 임의 생성

```
실행 순서:

1. 구현 전 체크리스트 출력 (코드 작성 전에 반드시)
   "⏸ HOOK-3: UI 구현 진입. 컴포넌트 확인 먼저 합니다."

2. 사용할 컴포넌트 목록 먼저 선언
   "이 태스크에서 사용할 컴포넌트:
    - [컴포넌트명]: src/lib/comps/[경로] — 실제 존재 확인 ✅/❌
    - [컴포넌트명]: src/lib/comps/[경로] — 실제 존재 확인 ✅/❌"

3. 존재하지 않는 컴포넌트 발견 시
   → 즉시 Stephen에게 보고
   "⚠️ [컴포넌트명]이 src/lib/comps/에 없습니다.
    기존 컴포넌트로 대체 가능한 것: [대안]
    새로 만들어야 하면 GATE B에서 태스크 추가 필요합니다."
   → 임의로 새 컴포넌트 생성하지 않고 대기

4. CSS 변수·클래스 사전 확인
   "사용할 스타일:
    - 색상: var(--[실제 변수명])
    - 타이포: .[실제 클래스명]
    - 간격: [실제 토큰값]"
   → ui-ux-standards.mdc 또는 ui-mobile.mdc에서 읽은 실제 값만 사용

5. 구현 완료 후 자가 체크
   "UI 자가 체크:
    [ ] 새 컴포넌트 생성 없음
    [ ] 하드코딩 색상 없음 (var(--) 사용)
    [ ] 직접 font-size 지정 없음 (클래스 사용)
    [ ] 터치 타겟 44px 이상"

금지:
❌ 컴포넌트 존재 확인 없이 import
❌ 없는 컴포넌트 임의 생성
❌ 하드코딩 색상값 (#ffffff, #333 등)
❌ 직접 font-size / font-weight 지정
```

## HOOK-4 — 새 채팅 재시작 후 첫 작업

> 패턴: 이전 세션 맥락 없이 시작해서 방향 이탈

```
새 채팅에서 TASK.md 로드 직후 아래를 반드시 실행한다.
TASK.md만 읽고 바로 구현 시작하지 않는다.

실행 순서:

1. 현재 상태 파악 (Read 툴)
   → TASK.md: DONE / NOW / NEXT / BLOCKED 현황
   → GSD_LOG.md: 마지막 완료 태스크와 파일
   → plan-output.md: 이번 사이클 아젠다 원문

2. 재시작 브리핑 출력
   "📋 재시작 브리핑
    아젠다: [plan-output.md 첫 줄]
    완료: [DONE 태스크 수]개
    현재 NOW: [태스크명]
    마지막 수정 파일: [GSD_LOG 마지막 항목]
    이어서 진행합니다."

3. NOW 태스크 도메인 판별 (STEP 1 — 기존 gsd-agent 로직)
   → TDD/GSD 모드 확인 후 진행

금지:
❌ TASK.md만 보고 바로 구현 시작
❌ GSD_LOG 확인 없이 "처음부터" 시작
❌ 이전 세션 내용을 "기억"으로 채우기
```

## 컨텍스트 롯 의심 신호 — Stephen이 감지 시 입력할 명령

```
"컨텍스트 체크"  →  gsd-agent가 HOOK-4 재시작 브리핑 실행
"파일 확인해"    →  HOOK-1 실행 (추측 없이 실제 파일 Read)
"로직 읽어"      →  HOOK-2 실행 (핵심 로직 파일 직접 Read)
"컴포넌트 확인"  →  HOOK-3 Step 2~3 실행 (존재 여부 확인)
```
